# CLAUDE.md — name-a-woman

A 60-second recall game: type names of women, each valid one drops a card; every woman ever
named by anyone flips a permanent global "discovered" bit. Matching runs **client-side** against
a prebuilt index (no search service); the only server is a tiny Node API + Postgres on imos-vm
that validates accepted guesses and owns the global counter. Full design:
`docs/superpowers/specs/2026-06-27-name-a-woman-architecture-design.md`.

## Development workflow (worktrees only)

All feature work happens in a git **worktree on its own branch** — never on `main`.

- **Never edit features on `main`** and **never switch this checkout off `main`**
  (`git checkout <other>` / `git switch` in the primary clone is off-limits). The primary
  checkout stays on `main` so parallel work never collides.
- Start each feature in its own worktree:
  `git worktree add ../name-a-woman-<feature> -b <feature>`. One worktree per feature, one
  branch per worktree.
- When done and verified, merge the branch back to `main` locally (fast-forward or `git merge`),
  then `git worktree remove`. Open a PR only if explicitly asked.
- **Visual verification happens on `main`, after merge.** The single local dev server runs only
  against the primary `main` checkout, so changes to eyeball (card animations, layout, the timer
  sweep) get merged first, then verified there — iterate on `main` if the visual pass finds
  issues. Build/typecheck/lint verification still happens in the worktree before merging.

## Carried principles (from influencer-tracker, adapted)

These are decided in the architecture spec; document the *why* here so they survive.

- **Self-host over managed services.** We dropped Supabase and chose not to add Typesense — the
  index ships as a static asset and matching is local. The server is self-hosted on imos-vm
  (`ssh ubuntu@imos-vm`, Docker). Prefer this to paid tiers whose costs surprise you (the sister
  project burned through a DB egress free-tier and had to revert — measure before relying).

- **One shared matching function, two runtimes.** `match()` is pure TypeScript imported by both
  the browser and the Node validator. Never reimplement matching server-side in a different
  engine — divergence (client accepts, server rejects) is a player-facing bug. Consistency beats
  a marginally better second implementation.

- **Validate at the trust boundary.** The client is untrusted; the server re-runs `match()`
  before flipping any global discovery bit. Client-side matching is a UX convenience, never the
  source of truth for the global counter.

- **Insert-only / frozen for reproducibility.** The `discovered` set is append-only
  (`INSERT ... ON CONFLICT DO NOTHING`); a woman's first-discovery is never rewritten. The shipped
  index is a timestamped monthly snapshot, regenerated by re-running the build job — never patched
  in place.

- **Least-privilege DB roles.** When the Postgres API is built, split roles: a public write path
  that can only INSERT into `discovered`/leaderboard, and a read path that is SELECT-only. No
  single connection string with owner rights in app code.

- **Fail-open on non-critical paths.** A counter-fetch or discovery-POST failure must never block
  the game — the player already has instant local feedback; the global write resolves in the
  background and degrades silently on error.

## Stack

Astro 7 (static, CDN) + Tailwind v4 + Biome + Bun. **No framework runtime** — the game is
plain HTML in `Game.astro` driven by a bundled client TS module (`src/game/game.client.ts`),
not React islands. The interactive surface (one input, flying cards, a counter) is
animation-heavy and imperative, so vanilla DOM + GSAP beats fighting a render cycle, and the
page ships zero framework JS. The pure logic (`match/`, `state`, `resolve`, `summary`, `global`,
`dev`) stays framework-agnostic and is imported by the script.

Animations: GSAP + Flip on the hot path (card fly-in, wall reflow, input dock) and the counter
tween; transitions.dev CSS for the card skeleton-reveal and the reject shake. **Do not add
astroanimate** (unmaintained, Astro 4/5 only, wrong paradigm — see spec §8).
